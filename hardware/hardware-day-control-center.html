1 · Host — bez urządzenia

Bezpieczny gate przed podłączeniem kostki. Nie wykonuje akcji urządzeniowej.

cd /path/to/radio-esp32
git status --short
npm run check

2 · H0 — po zgodzie T3

Identyfikacja i backup. Wstaw wyłącznie potwierdzony port `/dev/cu.*`.

export PORT=/dev/cu.REPLACE_ME
export ESP32_ALLOW_DEVICE_ACTION=1
scripts/core2-device-action.sh preflight
scripts/core2-device-action.sh backup

3 · H1 — po drugiej zgodzie

Pierwszy zapis. Skrypt sam ponawia host gate, build i weryfikuje backup.

export CONFIRM_FLASH=YES
scripts/core2-device-action.sh flash
Na co uważać

Warunki natychmiastowego STOP

Niejednoznaczny port lub flash

Jeśli pojawia się więcej urządzeń, inny chip albo flash nie ma 16 MiB — odłączamy i diagnozujemy. Nie zgadujemy.

Brak pełnego backupu

Nie ma zapisu bez dokładnie 16 777 216 bajtów, poprawnego SHA‑256 i prywatnego miejsca przechowania.

Reset, czarny ekran, zły touch

Powtarzalny reset, nieczytelny ekran, zły transform dotyku lub brak głośnika lokalnego zatrzymuje H1.

Dane prywatne w logach

Nie publikujemy MAC, BSSID, SSID, haseł, prywatnych endpointów ani obrazu fabrycznego. Log sanitizujemy przed eksportem.

Nieodwracalne zabezpieczenia

Na pierwszym bring-up nie palimy eFuse, nie włączamy secure boot ani flash encryption. To osobna decyzja po H0.

Dobry blocker też jest sukcesem

Jeśli H0/H1 zatrzyma się bez utraty obrazu fabrycznego i z precyzyjnym dowodem, sesja nadal jest bezpiecznie zakończona.

✓

Sukces pierwszej sesji

Pełny backup i jednoznaczny rollback H0 oraz zaliczony smoke H1 — albo precyzyjny, ograniczony blocker bez utraty fabrycznego obrazu. H2–H4 mogą być kolejnymi sesjami; nie skracamy endurance tylko po to, żeby szybciej powiedzieć „gotowe”.
